Overall Risk: MODERATE

Production infrastructure (bootstrap/ArgoCD/ExternalSecrets) follows strong security patterns with Workload Identity Federation and Secret Manager. Risk is concentrated in experiments/ -- prototype code and pipeline automation with security gaps typical of rapidly iterated experimental work. Two Konflux release pipeline issues (C-07, C-08) and wildcard RBAC (C-05) pose real risk if patterns propagate to production.

Top Priority Fixes

  1. Pin Konflux release pipeline image and task revision (C-07, C-08) -- active supply chain risk
  2. Scope kube-applier RBAC to specific resources (C-05) -- cluster-admin equivalent
  3. Add credential file patterns to root .gitignore (M-25) -- prevents accidental secret commits
  4. Add CODEOWNERS file (H-14) -- enforced review for critical paths

Critical Findings (8)

▶ Critical C-01 Pervasive subprocess shell=True with Unsanitized Config Values
File
experiments/ho-platform-none/install-ho-platform-none/common.py:174
Category
Application — Command Injection
Issue
CommandRunner.run() uses shell=True for all commands. Every installer step passes environment-sourced config values (PROJECT_ID, ZONE, WORKER_NODE_NAME, etc.) via f-strings into shell command strings without escaping or sanitization.
Impact
If any config value contains shell metacharacters, arbitrary commands execute as the running user. All 20+ installer step files are affected.
Remediation: Use subprocess.run() with list arguments instead of shell=True. Validate inputs with shlex.quote(). Add character-set validation in InstallConfig.__post_init__().
▶ Critical C-02 JWT Decoded Without Signature Verification
File
experiments/auth/phase2-poc/cloud-function/main.py:172-180
Category
Auth — Broken Authentication
Issue
decode_jwt_claims() splits the JWT on "." and base64-decodes the payload without cryptographic verification. The extracted email claim (line 238) is used for IAM role lookups and Kubernetes group membership decisions.
Impact
An attacker could craft a JWT with arbitrary claims to impersonate any user and gain elevated Kubernetes access (e.g., cluster-admin).
Remediation: Use google.oauth2.id_token.verify_oauth2_token() to cryptographically verify JWT signatures before trusting claims.
▶ Critical C-03 eval with User-Influenced Variables
File
experiments/auth/phase2-poc/cloud-function/deploy.sh:115
Category
Application — Command Injection
Issue
A gcloud command string is built via variable interpolation (including interactively-entered IDP_API_KEY from read at line 90), then executed via eval "$DEPLOY_CMD".
Impact
Shell metacharacters in the API key input could execute arbitrary commands as the running user.
Remediation: Build arguments in a Bash array and expand with "${args[@]}" instead of using eval.
▶ Critical C-04 --insecure-skip-tls-verify with Bearer Token Authentication
File
experiments/auth/phase2-poc/demo-oidc-flow.sh:59,69
Category
Auth — Insecure Transport
Issue
kubectl commands disable TLS verification (--insecure-skip-tls-verify) while simultaneously sending bearer tokens (--token=$IDP_TOKEN). Line 73 confirms this grants cluster-admin access.
Impact
A man-in-the-middle attacker can intercept the bearer token and gain cluster-admin access to the hosted cluster.
Remediation: Use --certificate-authority with the cluster CA certificate instead of skipping verification.
▶ Critical C-05 Wildcard RBAC ClusterRole for kube-applier
File
experiments/kube-applier-gcp/hack/manifests/rbac.yaml:6-8
Category
Kubernetes — Excessive Privileges
Issue
ClusterRole grants get, list, watch, create, update, patch, delete on apiGroups: ["*"] and resources: ["*"] — functionally equivalent to cluster-admin.
Impact
Any compromise of the kube-applier service account gives complete cluster control, including reading all secrets and modifying RBAC.
Remediation: Enumerate the specific resource types kube-applier manages. Remove wildcard apiGroups, resources, and verbs.
▶ Critical C-06 Cluster-wide Secret Read Access for Prometheus/GMP
Files
experiments/google-managed-prometheus/prometheus-cluster-wide.yaml:25-28, gmp-podmonitoring-example.yaml:24-26
Category
Kubernetes — Excessive Privileges
Issue
Two ClusterRoles grant get, list, watch on all secrets in all namespaces for Prometheus/GMP TLS scraping.
Impact
Compromise of Prometheus or GMP collector allows reading every secret cluster-wide: credentials, tokens, TLS keys, pull secrets across all namespaces.
Remediation: Use namespace-scoped Roles where possible. Add resourceNames to restrict which secrets are accessible.
▶ Critical C-07 Unpinned :latest Image in Konflux Release Pipeline
File
konflux/release-pipelines/tasks/push-snapshot-to-gar.yaml:40-41
Category
Supply Chain — Image Integrity
Issue
The production release task uses image: quay.io/jimd_openshift/release-service-utils-gcr:latest. A TODO acknowledges this. This is a production release pipeline pushing images to Google Artifact Registry.
Impact
Full supply chain compromise. A poisoned image from this personal Quay account could exfiltrate secrets, tamper with release artifacts, or inject backdoors into pushed images.
Remediation: Pin to a specific digest from an organizational registry. Complete the TODO to move to a Konflux-built image.
▶ Critical C-08 Konflux Pipeline Task Resolution Uses Mutable main Branch
File
konflux/release-pipelines/push-snapshot-to-gar.yaml:30-31
Category
Supply Chain — Pipeline Integrity
Issue
Pipeline resolves its task definition via git resolver with default: main. Task definitions can change between runs without any pipeline change.
Impact
A commit to main that modifies the task definition is automatically picked up by subsequent release pipeline runs, enabling supply chain attacks.
Remediation: Pin taskGitRevision to a specific commit SHA or release tag. Never use main as default for production release pipelines.

Positive Observations

  • ExternalSecrets properly configured — GCP Secret Manager via Workload Identity Federation

  • kube-applier-gcp scripts use set -euo pipefail consistently
  • Tekton provision pipeline has input validation step with allowlisting
